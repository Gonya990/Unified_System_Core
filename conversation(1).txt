The operation of distributed, non-federated agents coordinating in 
a common Git repository through **billboards, mailboxes, and 
beads** integrated with the **Ripple Effect Protocol (REP)** creates 
a hybrid architecture that balances **global visibility** with 
**tactical P2P alignment**.
### **1. The Billboard: Strategic Goal Decomposition**
The system treats the Git repository's central documentation (e.g., 
`spec.md` or `plan.md`) as a **Blackboard system** [1, 2].
*   **Knowledge Source Monitoring:** Specialized agents act as 
"Knowledge Sources" that monitor the billboard for subtasks that 
match their internal expertise [1, 2]. 
*   **LLM-Driven Planning:** A "master planner" agent (often 
utilizing a **puppeteer-style paradigm**) breaks down high-level 
business goals into bite-sized "tickets" on the billboard [3, 4]. 
*   **Global Context:** This billboard serves as a **dynamic library** 
of the current state, preventing agents from operating in silos and 
ensuring everyone is on the same page before code generation 
begins [2, 5].
### **2. The Mailboxes: Tactical P2P Negotiation**
Since agents are non-federated and potentially belong to different 
organizations, they use **standardized P2P protocols** (A2A, ACP, 
or ANP) to negotiate work before committing it [6, 7].
*   **Discovery via Agent Cards:** Agents find each other's 
mailboxes by reading "Agent Cards" or "Manifests" that detail 
capabilities and security requirements [8, 9].
*   **Protocol Routing:** A **ProtocolRouter** dynamically selects 
the best mailbox for the task: **ANP** for secure, identity-bound 
medical or financial data; **ACP** for high-throughput streaming 
tasks; or **A2A** for complex enterprise hand-offs [10-12].
*   **Maker-Checker Loops:** Agents use direct mail to implement 
**Group Chat orchestration**, where a "maker" proposes a solution 
and a "checker" provides an iterative critique until the result meets 
the quality threshold defined in the `spec.md` [13-15].
### **3. The Beads: Stigmergic Execution and Versioning**
Execution is recorded in the Git repo as a series of 
**"beads" (granular commits)** that serve as the environment's 
permanent state [16].
*   **Commits as Save Points:** Agents follow a "commit early and 
often" strategy, using commits as **save points in a game** to allow 
the system to revert or branch if an AI-generated refactor goes 
sideways [16].
*   **Isolation via Worktrees:** To scale massively parallel work, 
agents use **Git worktrees**, creating sandboxed environments for 
independent subtasks that are merged into the main line only after 
passing automated CI/CD quality gates [17, 18].
### **4. The Ripple: Autonomous Alignment via Textual 
Sensitivities**
The **Ripple Effect Protocol (REP)** is integrated into the "beads" 
to prevent the **bullwhip effect** in the repository [19, 20].
*   **Textual Sensitivities in Trailers:** When an agent pushes a 
commit (a bead), it includes a **lightweight textual sensitivity** in 
the Git commit trailer (metadata) [21, 22]. This signal explains how 
its decision would change if environmental variables (like a 
dependency's API or a database schema) shifted [23].
*   **Textual Gradient Synthesis:** Neighboring agents parse these 
trailers to perform a **generalized gradient step** [24]. Instead of 
just seeing *what* changed, they use LLM-based synthesis to 
understand *why* it changed and how they should align their own 
future moves to maintain stability [25, 26].
*   **Global Consensus Rules:** For group-wide decisions (e.g., 
selecting a new library), the system uses **deterministic consensus 
rules** like the coordinate-wise median to prevent local outliers 
from destabilizing the repo's architecture [27].
### **Summary of the Integrated Workflow**
| Stage | Component | Action |
| :--- | :--- | :--- |
| **Strategy** | **Billboard** | Master agent updates `spec.md`; 
specialists claim tasks [2, 4]. |
| **Negotiation**| **Mailbox** | Agents use **A2A/ACP** to refine the 
plan via **Maker-Checker** loops [10, 14]. |
| **Execution** | **Beads** | Agents push **granular commits** to 
isolated **worktrees** [16, 17]. |
| **Alignment** | **Ripple** | Agents share **textual sensitivities** in 
commit trailers to ripple alignment [21, 24]. |

**Analogy for Understanding:**
Think of this system like a **large-scale jazz ensemble** recording 
an album. The **billboard** is the lead sheet (the song structure) 
everyone sees on their stands. The **mailboxes** are the brief nods 
and whispered cues between the pianist and the bassist ("Let’s go 
double-time on the next chorus"). The **beads** are the actual 
notes recorded on the track—each one a permanent addition. 
Finally, the **ripple effect** is the way one player's **improvisational 
flexibility** affects the others: if the drummer hits the snare slightly 
harder (the signal), the others don't just copy him; they sense that 
"sensitivity" and shift their own volume to match the new energy, 
ensuring the whole band stays perfectly in sync without a 
conductor.

Cons / risks (likely failure modes)
1) Incentive + coordination overload (repo thrash)
A blackboard can devolve into high-frequency edits that constantly invalidate 
agent plans (“spec churn”).
“Commit early and often” can become noise early and often unless there are 
conventions for what qualifies as a bead vs a scratchpad.
2) “Textual sensitivities” may be underspecified and easy to game
If trailers aren’t schema-
validated, agents will write vague prose (“might change if APIs change”) that 
doesn’t support reliable downstream alignment.
LLM parsing/synthesis of trailers can introduce hallucinated causal links
unless grounded in concrete dependencies (files, symbols, versions, tests).
3) Consensus-by-median is not enough for architectural decisions
Coordinate-wise median works for numeric-ish vectors, but architecture is often 
non-convex (you can’t “median” between ORM vs raw SQL cleanly).
Deterministic rules help stability, but you still need legitimacy: who is allowed to 
decide, and how exceptions are handled.
4) Cross-org P2P protocols add operational complexity
Discovery (Agent Cards), identity, authN/Z, routing, and policy enforcement 
become a control plane of their own.
If mailbox negotiation is off-repo, you risk lost 
rationale unless distilled back into the billboard and/or commit trailers.
5) Security & supply-chain threat surface increases
Non-federated agents from different orgs pushing code implies:
higher risk of malicious commits, prompt injection in repo content, 
dependency poisoning
need for strong provenance, signing, policy-as-code, and automated 
review

The architectural model for distributed, non-federated agents can 
be significantly hardened by incorporating formal patterns from 
seminal software development texts. Translating concepts from 
**Domain-Driven Design (DDD)**, **Continuous Delivery (CD)**, and 
**Functional Thinking** ensures that the system moves from an 
"experimental swarm" to a rigorous, enterprise-grade engineering 
environment.
### **1. Domain-Driven Design (DDD) for Billboards and Context**
The **Billboard** (shared intent) should not merely be a task list; it 
must function as the repository for the **Ubiquitous Language** 
and **Strategic Distillation** of the system [1, 2].
*   **Bounded Contexts as Repositories:** Since these agents are 
non-federated (independently owned), each should operate within 
an explicit **Bounded Context** [3]. The Git repository should use 
directory structures to enforce these boundaries, ensuring that 
terms in a "Shipping Context" do not collide with an "Invoicing 
Context," even if they both use a "Charge" object [4, 5].
*   **Domain Vision Statement:** The master `spec.md` (the 
Billboard) should include a **Domain Vision Statement** [6]. This 
allows autonomous agents to prioritize the **Core Domain**—the 
most valuable part of the system—while relegating peripheral tasks 
to **Generic Subdomains** [7, 8].
*   **Aggregates for Bead Integrity:** When an agent places a 
"bead" (commit), it should respect **Aggregate** boundaries [9]. 
The agent should only modify the "Root Entity" of a cluster of 
objects to maintain transactional integrity, preventing different 
agents from "trampling" on internal data of the same complex 
structure [10].
###   **2. Continuous Delivery (CD) for Beads and Pipelines**
The **Beads** (granular commits) are the entries into the 
**Deployment Pipeline**, a paradigm for moving code from check-in 
to production in a controlled way [11-13].
*   **Release Candidates:** Every "bead" (commit) is technically a 
**Release Candidate** [14]. The Git repository must use 
**Continuous Integration (CI)** to immediately "disprove" the 
fitness of a bead through an automated **Commit Stage** [15, 16].
*   **Trunk-Based Development:** To prevent "integration hell," 
agents must follow **Trunk-Based Development**, keeping branch 
lifetimes to less than a day and merging into the mainline 
frequently [17, 18]. This ensures the "community memory" of the 
repository stays current [19].
*   **Automated Acceptance Test Gate:** Successful beads move to 
an **Acceptance Test Stage**, where the system validates that the 
agent’s contribution fulfills the business requirement defined on 
the Billboard [20, 21].
###   **3. Functional Thinking for Ripple Effect Protocol (REP)**
The **Ripple Effect Protocol** relies on sharing "textual 
sensitivities" (how a decision would change if variables shift) [22]. 
This maps directly to the distinction between **Actions, 
Calculations, and Data** [23].
*   **Sensitivities as Calculations:** A sensitivity signal should be 
treated as a **Calculation**—a pure function that takes 
environmental inputs and returns a "what-if" result without side 
effects [24, 25]. Because calculations are **referentially 
transparent**, other agents can aggregate these signals safely 
without triggering unintended changes in the system [26, 27].
*   **Timelines for Coordination:** Agents can use **Timeline 
Diagrams** to visualize how their P2P "mailbox" messages and Git 
"beads" interact over time [28]. This helps identify **race 
conditions** in distributed coordination, such as two agents 
attempting to refactor the same module simultaneously [29].
*   **Stratified Design:** Organize the repository in **Stratified 
Layers** [30]. The bottom layers contain "timeless" logic (e.g., 
basic math or Git utility functions), while the top layers contain 
"Business Rules" that change frequently, allowing agents to 
understand which parts of the repo are stable and which are 
"plastic" [31, 32].
###   **4. Summary Table: Translation into the Agent System**
| Agent System Component | Software Book Concept | Translation/
Implementation |
| :--- | :--- | :--- |
| **Billboard** | **Ubiquitous Language** [33] | Shared markdown 
files defining specific domain terms to prevent agent confusion. |
| **Billboard** | **Bounded Context** [3] | Explicit repo sub-
directories that isolate different agent team models. |
| **Mailbox** | **Service** [34] | P2P requests designed as stateless 
interfaces with no encapsulated state. |
| **Beads** | **Commit Stage** [16] | Automated CI triggered by 
every Git commit to validate code analysis and unit tests. |
| **Beads** | **Trunk-Based Dev** [17] | A rule that agents must push 
beads to the main branch at least once per day. |
| **Ripple (REP)** | **Side-Effect-Free Functions** [35] | Sensitivity 
signals must be pure logic, allowing safe aggregation by other 
agents. |

**Analogy for Understanding:**
Think of the system like a **highly automated modular factory**. 
**Domain-Driven Design** provides the **Standardized Blueprints** 
so that a motor made by one agent fits the chassis made by 
another. **Continuous Delivery** is the **Conveyor Belt (Pipeline)** 
that immediately scans every part for defects the second it’s placed 
(the Bead). **Functional Thinking** is the **Sensor Network** that 
doesn't just say a machine is "moving," but calculates its **Flexible 
Capacity**—allowing the entire floor to adjust its speed 
automatically if one robot slows down, without a human supervisor 
ever having to intervene.

#### **1. Billboard Canonical Source: Hybrid**
The sources suggest a **hybrid approach** is most robust. The 
**repository files** (e.g., `plan.md` or a `tasks/` directory) should be

#### **2. Task Structure: One-file-per-task**
A **one-file-per-task** structure (e.g., `tasks/uuid-slug.md`) is 
superior for distributed agents. This mirrors **"microlevel 
decomposition,"** where discrete tasks are handled by specialized 
agents [14, 15]. Technically, this minimizes **"trampling on the 
chalk"** (merge conflicts) on a single ledger file, which is a known 
bottleneck in high-scale blackboard systems [16]. It also allows 
agents to "lease" specific files to signal active work, preventing 
redundant effort.
#### **3. Acceptance Criteria: Machine-Checkable by Default**
Criteria should be **machine-checkable (executable 
specifications)** by default to enable the "derivation of Done" from 
a green CI [17, 18]. Following the **"given-when-then" model** 
allows the system to verify the **"executable bedrock"** of a claim 
before merge [19, 20]. Human-only criteria should be reserved for 
high-level **qualitative sign-offs** (e.g., ethics, brand alignment, or 
strategic UX) [21, 22].

The development of a **Shared Intent** domain within an 
architecture R&D map focuses on creating a "community memory" 
that allows autonomous agents to coordinate without centralized, 
rigid orchestration. Drawing on the sources, here is a high-level 
map for this domain:
### Domain A: Shared Intent (Billboard / Blackboard)
The goal of this domain is to replace static, request-response 
interactions with **contextual continuity** and **emergent 
collaboration** [1]. This is achieved through a **Blackboard 
System**, defined as a common knowledge base iteratively updated 
by diverse specialist "knowledge sources" (agents) to solve 
complex, ill-defined problems [2, 3].
#### 1. The Minimum Shared State
To avoid siloed work, agents must have visibility into a "global 
workspace" that transcends individual sessions [3, 4]. The 
minimum shared state includes:
*   **Goal Specifications and Tasks:** High-level objectives (e.g., 
"Prepare a market analysis") that are decomposed into subtasks 
for execution [2, 5].
*   **Partial Solutions and Suggestions:** Intermediate results 
"published" to the board that enable other specialists to apply their 
expertise [3, 4].
*   **Environmental Traces (Stigmergy):** In stigmergic 
coordination, the "trace" left in the environment by one agent's 
action stimulates the next action by another agent [6]. This serves 
as a **shared external memory** [7].
*   **Shared World Representations:** Common ontologies (e.g., 
OWL-S) and knowledge graphs that ensure all participants share a 
machine-understandable vocabulary for goals and constraints [8, 
9].
#### 2. Preventing "Board Noise" and Plan Churn
Constant churn and cognitive overload are significant risks in high-
velocity multi-agent systems [10]. Prevention strategies include:
*   **The Control Shell:** Acting as a moderator, the control shell 
organizes the use of agents to prevent them from "trampling each 
other" or creating conflicting updates [3].
• 
*   **Effective Memory Management:** Implementing mechanisms 
for **selective retention** and **contextual summarization** to 
prevent the accumulation of excessive, irrelevant context [10].
*   **Knowledge Distillation:** Actively curating the shared state to 
compress memories into compact policies and resolve behavioral 
conflicts, thereby avoiding **knowledge entropy** [11].
*   **Plan Traceability and Versioning:** Maintaining versioned 
workflows and audit trails allows the system to roll back ineffective 
plans rather than allowing them to churn indefinitely [12, 13].
#### 3. Billboard (Public) vs. Private Negotiation
A critical architectural distinction must be made between what is 
broadcast to the community and what stays in local agent 
negotiation:
*   **On the Billboard (Public):** 
    *   **Intent Exchange:** High-level goals and "requests for 
volunteers" where agents can bid based on their specific partitions 
of expertise [14, 15].
    *   **Committed Intentions:** Actions to which an agent has 
officially committed within a BDI (Belief-Desire-Intention) 
framework [16, 17].
    *   **Shared Norms and Protocols:** The "social contracts" or 
behavioral schemas (like turn-taking or conflict resolution 
strategies) that govern the society [18, 19].
*   **In Private Negotiation:** 
    *   **Internal Mental States:** An agent's private "Beliefs" and 
"Desires" remain internal until they translate into a public 
commitment [16, 17].
    *   **Bidding and Role Negotiation:** Initial "propose-justify-
revise" cycles between agents occur via direct dialogue before a 
final coalition or contract is posted to the board [20-22].
    *   **Uncertainty Modeling:** Internal confidence scores or 
Bayesian reasoning trajectories stay private to avoid polluting the 
board with low-confidence noise [23, 24].
***
**The Specialist Metaphor**
To understand this architecture, imagine a **group of specialists in 
a room with a large blackboard** [4]. The session begins with a 
problem written on the board. The specialists do not talk to each 
other directly; instead, they watch the board for an opportunity to 
apply their specific expertise. When one expert writes a partial 
solution, it acts as a "stigma" (a sign) that triggers another expert 
to step forward and add the next piece. The board remains clean 
because a **moderator** (the control shell) ensures only relevant, 
high-quality information is recorded, while the experts' internal 
"thinking" and "debating" happen quietly at their desks until they 
are ready to contribute [3, 4].

Beads (change units) in multi-agent work — pros, cons, 
constraints, mechanisms
I’ll treat “bead” as: the smallest auditable execution unit that changes repo state, 
typically a commit and/or a PR that packages one or more commits.
1) Pros (why beads work well for multi-agent 
systems)
A) Concurrency with low coordination overhead
Agents can work in parallel on separate beads with minimal coupling.
Enables “stigmergy”: a merged bead becomes a trace that other agents can 
react to.
B) Strong auditability + reproducibility
Each bead is a durable record: who changed what, when, why (if metadata is 
enforced).
Supports rollback, blame, provenance, and post-incident analysis.
C) Natural gating surface
Beads can be forced through repeatable gates (lint/tests/policy/supply-chain), 
making quality scalable to many agents.
Encourages a “trust, but verify” automation model (good for agent 
contributions).
D) Composability into higher-order outcomes
Many small beads compose into a task outcome, while still allowing intermediate 
stabilization points.
2) Cons / risks (what breaks in practice)

A) Fragmentation (too many micro-beads)
Excessively small beads create:
review fatigue
noisy history
harder reasoning about “what changed logically”
Multi-agent systems tend to over-produce “near-duplicate” beads unless 
constrained.
B) Hidden coupling → merge conflict storms
Agents don’t “feel” global coupling until merge time:
same files/configs touched
shared contracts/schemas
shared CI pipelines
Without coupling signals, you get thrash (reverts + rework loops).
C) CI gaming / minimum-gate overfitting
If arbitration is “first-to-green-min-gate,” agents may optimize for passing the 
minimal gate, not for correctness or long-term maintainability.
D) Incentive misalignment
“Win the lease” can become the objective.
Some agents may hold the system hostage via:
slow PRs
frequent small updates to keep lease alive
“looks green” changes that increase latent risk.
E) Security surface
Each bead is an injection point:
malicious diffs
prompt-injection text into repo docs/norms
dependency poisoning
Signed commits/provenance help, but don’t eliminate semantic poisoning.

Following the handoff from **Domain A (Shared Intent)**, the design 
for **Domain B (Execution / Change Units)** focuses on the 
decentralized mechanisms that transform committed intentions 
into verified system truth.
### (a) Primitives: The "Increment" or "Delta"
The conceptual unit of execution is the **Change Unit** (shortlisted 
as **Increment** or **Delta**), which acts as the atomic bridge 
between a public task-claim and a potential update to the system's 
world state [1, 2].
*   **Encapsulation:** It must contain a **Proposal** (the intended 
modification) and a **Validation Pointer** (links to evidence that it 
meets project constraints) [3, 4].
*   **Stigmergic Function:** Every Change Unit serves as a "trace" 
or **stigma** in the environment, which stimulates and guides the 
succeeding actions of other specialized agents without requiring a 
central coordinator [5, 6].
*   **Reasoning Isolation:** To prevent board noise, the **Chain-of-
Thought (CoT) traces** remain in the agent’s private workspace;
